# 智能提交 — 自动判断提交策略

根据工作区状态自动选择提交路径：
- **暂存区有内容** → 路径 A：分析暂存区，生成 message，提交
- **暂存区为空** → 路径 B：分析全部工作区变更，拆分提交

## 提交规范

项目使用 Conventional Commits 格式：

```
<type>(<scope>): <简短描述>

- 具体变更点1
- 具体变更点2
```

**type 取值**：
- `feat` — 新功能
- `fix` — 修复 bug
- `refactor` — 重构（不改变外部行为）
- `docs` — 文档
- `chore` — 构建/工具

**scope 取值**（从变更文件路径推断）：
- `FuncDesign` — `src/components/FuncDesign/` 下的文件
- `ColumnDesign` — `src/components/ColumnDesign/` 下的文件
- `dynamicModel` — `src/views/common/dynamicModel/` 下的文件
- `FormGenerator` — `src/components/FormGenerator/` 下的文件
- 其他模块按目录名取

## 路径 A：直接提交（暂存区已有内容 / 单主题变更）

### A1: 收集信息

并行执行：

```bash
git diff --cached --stat      # 暂存区变更统计
git diff --cached              # 暂存区完整 diff
git diff --stat HEAD           # 工作区全量统计（用于判断是否有未暂存变更）
git log --oneline -5           # 近期提交风格
```

如果暂存区有内容，走路径 A。暂存区为空则走路径 B。

### A2: 分析变更

从 diff 中提取：

1. **涉及的模块**（从文件路径推断 scope）
2. **变更类型**（新增 / 修改 / 修复 / 重构）
3. **核心改动点**（每个文件改了什么）

### A3: 分析并拆分子任务

从 diff 中识别**功能关注点**（而非按文件罗列）。一个 commit 可能包含多个子任务，需要主动拆分。

**拆分方法**：

1. 逐文件阅读 diff，识别每个文件中的**功能意图**（新增能力 / 修改行为 / 修复问题 / 重构）
2. 将跨文件但属于同一功能意图的变更归为同一个子任务
3. 如果整个改动只围绕一个意图，标题行就够了，无需正文
4. 如果涉及多个意图，每个意图生成一个子任务
5. 子任务之间按依赖排序：基础变更在前，依赖它的在上

### A4: 生成 commit message

**标题行**：`<type>(<scope>): <一句话概括>`，不超过 72 字符。如果涉及多个 scope，选择最主要的。

**正文**：每个子任务一行，`- ` 开头。描述"做了什么功能"，让查看 log 的人能快速理解改动意图。

**子任务描述规范**：
- 一行一个子任务，描述功能意图而非实现细节
- 子任务之间有逻辑顺序（基础 → 扩展 → 集成）
- 如果子任务之间有因果关系，在子任务描述中体现"为 XX 提供 YY"

**小变更示例**（无需拆子任务）：
```
fix(FuncDesign): 修复分支条件数据源选择不回显的问题
```

**中变更示例**（2~4 个子任务）：
```
feat(dynamicModel): 批量按钮接入流程引擎

- 批量操作按钮支持触发流程，传入选中记录集合
- 流程数据上下文新增记录集合字段
- 循环节点自动识别批量数据并进入遍历模式
```

**大变更示例**（多个子任务，体现依赖关系）：
```
feat(FuncDesign): 弹窗赋值节点 + 接口/对话框数据源增强

- 新增弹窗赋值节点，支持将弹窗表单数据回填到主表单
- 接口节点支持将响应数据作为后续节点的数据源
- 对话框节点新增数据源配置，支持传入和传出数据
- 数据源选择器新增弹窗字段和接口响应的选项
```

### A5: 提交

将生成的 commit message 提交给用户审阅，确认后执行：

```bash
git commit -m "标题

- 变更点1
- 变更点2"
```

## 路径 B：拆分提交（多主题混合变更）

当工作区有大量混合变更时，自动进入此模式。

### 核心原则

1. **每个文件只归属一个提交** — 避免 `git add -p` 手动拆 hunk 的风险
2. **自底向上排序** — 叶子节点（无内部依赖）在前，最终整合在后
3. **每个提交可独立着陆** — 不依赖后续提交即可编译通过、功能完整
4. **保持内聚** — 同一提交内的变更围绕同一关注点

### B1: 读取全貌

并行执行：

```bash
git diff --stat HEAD        # 变更文件统计
git diff HEAD               # 完整 diff 内容
git status --short          # 工作区状态
git log --oneline -10       # 近期提交风格参考
```

如果涉及新文件，必须读取新文件完整内容。

### B2: 主题拆解

逐文件阅读 diff，按**关注线**分类：

| 类型 | 特征 |
|------|------|
| 基建/工具 | .gitignore、polyfill、配置变更，与业务无关 |
| Bug 修复 | 修正已有逻辑的错误行为 |
| 新功能 | 新增能力，有独立用户价值 |
| 增强 | 对已有功能的扩展或改进 |
| 重构 | 不改变行为的代码结构调整 |

输出主题表：每个主题列出涉及文件和一行摘要。

### B3: 依赖分析

对每个主题，识别：

- **内部依赖**: 该主题内文件的引用关系（A import B）
- **跨主题依赖**: 主题之间的数据/接口依赖
- **共享文件**: 被多个主题同时修改的文件

关键检查点：
- 新文件是否被其他文件 import → 新文件必须先于消费者提交，或在同一提交
- 函数签名变更 → 调用方必须在同一提交或已兼容
- props/emits 新增 → 父子组件需协调

### B4: 边界调整（解决共享文件问题）

**核心规则：一个文件只能出现在一个提交中。**

当多个主题修改同一文件时：

**策略 A: 归入最重的主题** — 共享文件完整变更归入依赖最深的主题
**策略 B: 合并主题** — 共享文件占比高、逻辑耦合紧时合并为一个提交

选择依据：哪个主题变更占比更高 → 归入该主题；合并后仍内聚 → 合并。

### B5: 排序

按依赖关系从叶子到根排序：
1. 无依赖的基建变更
2. 无跨提交依赖的独立修复/新组件
3. 核心集成（依赖前面的叶子）
4. 最终整合/UI 增强

### B6: 输出方案

按以下格式输出每个提交。每个 commit 的 message 同样遵循 A3/A4 的子任务拆分规则：

```
Commit N: `<type>(<scope>): <description>`

子任务：
- 子任务1描述
- 子任务2描述

文件：
- `file1.vue` — 一行说明
- `file2.ts` — 一行说明
```

同时输出每个提交的**独立着陆验证**和**依赖链说明**。

等待用户确认方案后再进入 B7。

### B7: 执行提交

方案确认后，按排序逐个执行：

1. **暂存文件**：`git add <该提交涉及的所有文件>`
2. **展示计划**：显示文件列表和生成的 commit message
3. **用户确认**：等待用户确认
4. **执行提交**：
   ```bash
   git commit -m "type(scope): 简短描述

   - 具体变更点1
   - 具体变更点2"
   ```
5. **验证**：`git status` 确认成功，继续下一个

**中断处理**：提交失败时暂停报告错误，修复后可从中断点继续。

## 严格禁止

- **禁止附加** `Co-Authored-By`、`Signed-off-by` 等任何署名行
- **禁止使用** `--no-verify` 跳过 hooks
- **禁止** `--amend` 修改已有提交（除非用户明确要求）
- **禁止** 一次性 `git add -A` 或 `git add .`（路径 B 中逐个提交按文件列表 add）

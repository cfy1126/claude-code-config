# 每日代码变更审查

对工作区中所有未提交的改动（`git diff` + untracked files）执行结构化审查，输出审查报告。

## 执行流程

### 第零步：理解意图

在执行任何检查之前，先阅读完整 diff，回答以下问题：
1. 本次改动的目的是什么？（一句话概括）
2. 涉及哪些模块/功能领域？
3. 改动的复杂度如何？（小/中/大）

如果无法理解意图（如 diff 过于碎片化），向用户确认后再继续。

### 第一步：收集变更信息

```bash
git diff --stat
git status -s
```

如果没有任何改动，直接告知用户"工作区干净，无需审查"并结束。

对每个变更文件，运行 `git diff <file>` 或读取 untracked 文件内容。

### 第二步：自动检查（工具执行）

按顺序执行以下检查，收集结果：

#### 2.1 ESLint 检查

**格式化规则**：ESLint `--fix` 自动修复**仅对新建文件**执行。修改已有文件时**不做格式化**（不传 `--fix`），避免产生与功能无关的格式变更 diff。

```bash
# 新建文件（untracked）：可以 --fix
npx eslint --no-ignore --fix <new_files>

# 修改已有文件（modified）：仅检查，不自动修复
npx eslint --no-ignore <modified_files>
```
- 报告：错误数、警告数、涉及的文件和行号
- 仅关注本次改动引入的新错误（如果文件有大量已有错误，明确标注"已有错误"和"新增错误"）

#### 2.2 类型检查
```bash
npx vue-tsc --noEmit 2>&1
```
- 报告：类型错误列表
- 仅关注本次改动文件相关的类型错误

#### 2.3 安全扫描（人工审查 diff）
逐文件审查 diff，检查以下安全问题：
- [ ] **XSS**：是否有 `v-html`、`innerHTML`、`dangerouslySetInnerHTML` 使用未过滤的用户输入
- [ ] **注入**：是否有拼接 SQL/命令/URL 的字符串（应使用参数化查询）
- [ ] **敏感信息**：是否有硬编码的密钥、token、密码、API Key
- [ ] **defHttp 使用**：API 调用是否走 `defHttp`（`/@/utils/http/axios/`），禁止直接使用 axios
- [ ] **any 类型**：是否有新增的 `any` 类型声明（项目禁止）

#### 2.4 潜在 Bug 审查（人工审查 diff）
逐文件审查 diff，检查以下问题：
- [ ] **空值处理**：可选链 `?.` 是否遗漏，数组/对象访问前是否检查存在性
- [ ] **类型安全**：`==` vs `===` 使用是否合理（项目中 jnpf-select 值为字符串，某些场景需 `==`）
- [ ] **响应式丢失**：`ref`/`reactive` 解包是否正确，`toRaw` 使用场景是否必要
- [ ] **内存泄漏**：事件监听器/定时器是否在 `onUnmounted` 中清理
- [ ] **异步竞态**：多个异步操作是否存在竞态条件（快速连续触发）
- [ ] **死代码**：是否有未使用的变量、import、函数

#### 2.5 规范一致性审查
- [ ] **import 路径**：是否使用 `/@/` 和 `/#/` 别名，禁止相对路径深层次 import
- [ ] **组件写法**：是否使用 `<script setup lang="ts">` + Composition API
- [ ] **API 归档**：新增的 API 调用是否放在 `src/api/` 对应业务域文件中
- [ ] **store 使用**：Pinia store 是否按领域拆分，是否有跨模块状态混入
- [ ] **命名风格**：变量/函数命名是否符合项目现有风格（camelCase）

### 第三步：输出审查报告

按以下格式输出结构化报告：

```
## 代码审查报告

### 意图理解
本次改动目的：<一句话概括>
涉及模块：<模块列表>
复杂度：<小/中/大>

### 变更概览
- 改动文件：N 个
- 新增文件：N 个
- 总增/删行数：+N / -N

### 自动检查结果

#### ESLint: PASS / FAIL (N 个错误)
- （如有错误，列出文件和行号）

#### 类型检查: PASS / FAIL (N 个错误)
- （如有错误，列出文件和行号）

### 详细审查

#### [M-001] <问题标题> (Must fix)
- **位置**：`file.vue:行号`
- **问题**：<具体描述>
- **建议修复**：<具体方案>

#### [S-001] <问题标题> (Should fix)
- **位置**：`file.vue:行号`
- **问题**：<具体描述>
- **建议修复**：<具体方案>

#### [N-001] <建议标题> (Nice to have)
- **位置**：`file.vue:行号`
- **建议**：<优化建议>

#### <审查通过的区域> (Looks good)

### Verdict: PASS / CAUTION / BLOCK
- Must fix：N 项（列出编号）
- Should fix：N 项（列出编号）
- Nice to have：N 项
- Looks good：N 项
- 裁定：<PASS 可以提交 / CAUTION 建议修复后提交 / BLOCK 必须修复>
```

## 注意事项

- 审查仅关注本次改动引入的问题，不审查文件中的已有问题
- 如果某个检查项不适用于当前改动类型（如纯样式修改不需要安全审查），标注"不适用"并跳过
- 使用中文输出报告
- 对于发现的问题，给出具体的修复建议（指出文件路径和行号），不要只说"有问题"
- 审查发现的每个问题必须编号（M-001, S-001, N-001 格式），便于后续跟踪和确认修复
- Verdict 为 BLOCK 时，建议用户修复后重新运行 `/code-review`
- Verdict 为 CAUTION 时，由用户决定是否立即修复
- 如果变更仅涉及非代码文件（如 .md、配置文件），可简化审查流程

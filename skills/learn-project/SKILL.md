# 学习项目 — 全面架构分析

系统性阅读项目源码，按主次分步骤输出架构说明文档。

## 执行步骤

### 第一步：项目全貌扫描

1. 列出项目所有源文件（排除 node_modules、.git、dist、__pycache__ 等）
2. 读取 `package.json` 或等效的项目配置文件，确定技术栈和项目类型

### 第二步：按主次逐层阅读

严格按以下优先级顺序阅读源码，**不要跳过任何层级**：

**第一层：入口与引导**
- 应用入口文件（main.js / index.js / App.vue / main.ts 等）
- 公共配置（public-path、环境变量、构建配置）

**第二层：核心基础设施**
- 路由配置
- 状态管理（Store / Context）
- 工具函数和桥接层
- API / 请求封装

**第三层：布局与导航**
- 根组件和布局组件
- 页面注册表 / 导航配置

**第四层：业务页面**
- 按功能模块逐个阅读页面组件
- 每个页面的子组件

**第五层：样式与资源**
- 全局样式、主题配置
- 静态资源

### 第三步：生成架构文档

按以下结构输出分析结果，**所有代码引用必须标注文件路径**：

```markdown
# [项目名] 项目架构说明

## 一、项目定位
（技术栈、项目类型、一句话定位）

## 二、核心架构（按主次排列）

### 1. [最核心的机制] — `文件路径`
（解释关键逻辑）

### 2. [次核心的机制] — `文件路径`
（解释关键逻辑）

...

## 三、业务页面

### N. [页面名] — `文件路径`
（功能描述、组件职责表、数据流）

## 四、数据流总结
（ASCII 图示或文字链路）

## 五、文件结构
（树形目录）
```

**编写原则**：
- 每个模块标题标注对应的文件路径
- 使用表格列出函数/组件/配置的职责
- 数据流用简洁的 ASCII 图示
- 不要堆砌代码，用自然语言解释"为什么"和"怎么运转"

### 第四步：保存文档

1. 确认项目 `doc/` 目录存在，不存在则创建
2. 文件名格式：`doc/project-YYYY-MM-DD.md`（取当天日期）
3. 如果当天文件已存在，追加后缀：`doc/project-YYYY-MM-DD-v2.md`
4. 输出文件路径确认

## 写作风格要求

**核心目标**：让刚接触项目、不了解业务的开发者也能看懂。用"打比方"代替术语堆砌，用"实际例子"代替抽象描述。

### 1. 用生活类比解释技术概念

遇到技术术语时，先给一个生活中的类比，再说技术细节：

```
❌ "Vuex Store 采用模块化设计，通过 factory function 返回初始状态"
✅ "Vuex Store 就像一个共享的白板，所有页面都能在上面读数据和改数据。
    每次切换场景时，白板会被擦干净重写（resetScene），防止上一个页面的数据残留"
```

### 2. 用"如果你要..."的句式讲清职责

每个核心模块，用开发者实际会遇到的问题来解释它负责什么：

```
❌ "MapController 负责覆盖物 CRUD、弹窗、高亮、定位等操作"
✅ "MapController 就像地图的遥控器：
    - 如果你要在地图上画一个点 → 调 addMarker()
    - 如果你要点击一个点弹出详情 → 调 showPopup()
    - 如果你要让某个区域高亮闪烁 → 调 highlight()
    - 如果你要让镜头飞到某个位置 → 调 flyTo()"
```

### 3. 用"从头到尾"的流程串起数据流

不要只画抽象架构图，用一条完整的业务链路讲清楚"用户做了什么 → 系统怎么响应 → 结果是什么"：

```
❌ "数据流：config.js → initScene → getters → MapController → AMap"
✅ "用户打开页面后，数据是这样流动的：
    1. 页面加载 → 读取 config.js（告诉系统：这个页面有哪些图层、用什么筛选器）
    2. initScene → 把配置和数据灌进 Vuex（共享白板写好了）
    3. 筛选器变化 → FilterEngine 过滤数据 → filteredPoints 自动更新
    4. Vue 监听 filteredPoints 变化 → 调 MapController 同步地图上的标记点
    5. 用户点击标记 → MapController 弹出 InfoWindow 显示详情"
```

### 4. 函数/组件职责表用"一句话人话"描述

表格里的"职责"列不要写技术实现，写"它帮我们做了什么"：

```
❌
| 函数 | 职责 |
|------|------|
| initScene | 初始化场景 state，重置所有模块数据 |

✅
| 函数 | 它帮我们做了什么 |
|------|------|
| initScene | 场景切换时"清场"——把上一个页面的地图数据、筛选条件、弹窗全部清空，再装入新页面的配置 |
```

### 5. 文件结构树加"一句话注释"

目录树里每个关键文件夹/文件后面加一句简短注释，让新人一眼知道它干什么：

```
❌
src/
├── components/
│   └── map/

✅
src/
├── components/
│   └── map/          ← 通用地图"引擎"，所有地图页面共用这一套
│       ├── MapController.js   ← 地图遥控器：画点、画线、弹窗、高亮都在这里
│       ├── AMapContainer.vue  ← 地图容器：负责创建地图实例，把遥控器交给页面用
│       └── panels/            ← 左右两侧的面板组件（筛选、详情、统计等）
```

### 6. 避免的写法

- 不用"该模块实现了 xxx 能力"这种被动句式，用"这个模块帮你做 xxx"
- 不连续使用超过 3 个技术术语而不解释
- 不假设读者了解项目背景（如"和主应用的对接"要解释清楚主应用是什么）
- 不写"详见源码"这种无信息量的句子
